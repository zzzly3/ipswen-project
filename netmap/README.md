# netmap —— 高性能数据包 I/O 框架与 VALE 软件交换机 / netmap — High-Performance Packet I/O Framework and VALE Software Switch

## 概述 / Overview

`netmap` 是一个用于超高速用户态数据包 I/O 的框架，配合 **VALE**（一个基于 netmap API 的内核级 L2 软件交换机），可在单个进程内处理**每秒数千万个数据包**，匹配 10Gbps 和 40Gbps 端口的线速（即使是最小帧）。

`netmap` is a framework for ultra-high-speed userspace packet I/O, combined with **VALE** (a kernel-level L2 software switch based on the netmap API), capable of processing **tens of millions of packets per second** in a single process, matching line rate on 10Gbps and 40Gbps ports (even with minimum-sized frames).

本仓库是 IPswen 项目环境中使用的 netmap 本地副本，包含 **Linux 内核模块**、**VALE 交换机**、**修改版 Intel/NVIDIA/Realtek 网卡驱动**（强制启用 netmap 感知）、以及完整的用户态编程 API。

This repository is a local copy of netmap used in the IPswen project environment, including the **Linux kernel module**, **VALE switch**, **patched Intel/NVIDIA/Realtek NIC drivers** (with netmap awareness forcibly enabled), and a complete userspace programming API.

**核心定位：** netmap 是 IPswen 实验平台的**底层高性能数据面**——其他组件（如 optmap 的 XDP 路由、fakedns 的地址转换、happyfootball 的多路径连接）运行在 netmap 之上的用户态协议栈中，netmap 负责在最底层以最高速度将原始数据包从网卡送到用户态应用，或从用户态应用发送到线路上。

**Core positioning:** netmap serves as the **low-level high-performance data plane** for the IPswen experimental platform — other components (such as optmap's XDP routing, fakedns's address translation, happyfootball's multipath connections) run in the userspace protocol stack above netmap, while netmap is responsible for delivering raw packets from the NIC to userspace applications (or from userspace applications to the wire) at maximum speed at the lowest layer.

**为什么在 IPswen 项目中使用 netmap（而非标准 Linux socket）：**

**Why use netmap in the IPswen project (instead of standard Linux sockets):**

| 特性 / Feature | 标准 Linux Socket / Standard Linux Socket | Netmap |
|------|-------------------|--------|
| 零拷贝 / Zero-copy | 需要在用户态/内核态间拷贝 / Requires copying between userspace and kernelspace | 内存映射，零拷贝 / Memory-mapped, zero-copy |
| 批量 I/O / Batch I/O | 每次系统调用 1 个数据包 / 1 packet per syscall | 每次系统调用最多 1024 个数据包 / Up to 1024 packets per syscall |
| 每包系统调用数 / Syscalls per packet | ~1 | ~0.001（批处理）/ ~0.001 (batching) |
| 缓冲区管理 / Buffer management | sk_buff 动态分配/释放 / sk_buff dynamic alloc/free | 预分配环形缓冲区 / Pre-allocated ring buffers |
| 吞吐量（64B 小包）/ Throughput (64B packets) | ~1-2 Mpps | ~14.88 Mpps（10G 线速）/ ~14.88 Mpps (10G line rate) |
| CPU 利用率 / CPU utilization | 高（数据拷贝 + 系统调用开销）/ High (data copy + syscall overhead) | 低（零拷贝 + 批处理）/ Low (zero-copy + batching) |
| API 兼容性 / API compatibility | POSIX socket | 自定义 netmap API / libpcap 模拟 / Custom netmap API / libpcap emulation |

### 四个核心优化技术（按重要性排序） / Four Core Optimization Techniques (By Importance)

1. **I/O 批处理** —— 一次系统调用传输多个数据包（通常 512+），将系统调用开销分摊到每个包上 / **I/O Batching** — Transfers multiple packets (typically 512+) in a single syscall, amortizing syscall overhead across packets
2. **高效设备驱动** —— 驱动直接操作 netmap 的预分配环形缓冲区，绕过 sk_buff 的分配/释放 / **Efficient Device Driver** — The driver operates directly on netmap's pre-allocated ring buffers, bypassing sk_buff alloc/free
3. **预分配缓冲区** —— TX/RX 描述符环在打开时一次性分配（`NIOCREGIF`），数据路径中零动态内存分配 / **Pre-allocated Buffers** — TX/RX descriptor rings are allocated once at open time (`NIOCREGIF`), zero dynamic memory allocation in the data path
4. **内存映射缓冲区** —— 通过 `mmap()` 将内核环形缓冲区直接映射到用户态，用户程序直接读写数据包，零拷贝 / **Memory-Mapped Buffers** — Kernel ring buffers are directly mapped to userspace via `mmap()`, user programs read/write packets directly, zero-copy

**关键纠正：** 尽管叫"netmap"，内存映射（mmap）并不是最关键的性能优化。所有四个技术必须结合使用才能真正达到 10G/40G 线速。

**Key correction:** Despite the name "netmap", memory mapping (mmap) is not the most critical performance optimization. All four techniques must be used together to truly achieve 10G/40G line rate.

### 数据包收发流程 / Packet Transmit/Receive Flow

#### 接收流程 / Receive Flow
```
1. 硬件 DMA 将数据包写入预分配的 RX 缓冲区 / Hardware DMA writes packets into pre-allocated RX buffers
2. 驱动（netmap-aware）→ 更新 netmap RX ring 的 slot 描述符 / Driver (netmap-aware) → updates netmap RX ring slot descriptors
3. 用户态通过 ioctl(NIOCTXSYNC) 或 select/poll 发现新数据包 / Userspace discovers new packets via ioctl(NIOCTXSYNC) or select/poll
4. 用户态直接通过 mmap 映射的内存读取数据包内容（零拷贝）/ Userspace reads packet content directly via mmap'd memory (zero-copy)
5. 处理完成后更新 slot 状态，确认已消费 / After processing, update slot status to acknowledge consumption
```

#### 发送流程 / Transmit Flow
```
1. 用户态通过 mmap 映射的内存填写数据包内容 / Userspace fills packet content via mmap'd memory
2. 更新 netmap TX ring 的 slot 描述符（设置 buflen 等）/ Update netmap TX ring slot descriptors (set buflen, etc.)
3. 用户态调用 ioctl(NIOCTXSYNC) 通知内核有数据要发送 / Userspace calls ioctl(NIOCTXSYNC) to notify kernel of pending data
4. 驱动从 TX ring 读取 slot，DMA 到硬件发送队列 / Driver reads slot from TX ring, DMAs to hardware transmit queue
5. 硬件发送完成后，驱动更新 slot 状态（如 tx completion）/ After hardware transmission completes, driver updates slot status (e.g., tx completion)
```

## 子系统完整解析 / Complete Subsystem Breakdown

Netmap 的模块化设计包含以下子系统，通过 configure 的 `--enable-*`/`--disable-*` 选项控制编译：

Netmap's modular design includes the following subsystems, controlled via the `--enable-*`/`--disable-*` options of configure:

### 1. VALE（默认启用）—— 内核级 L2 软件交换机 / VALE (Enabled by Default) — Kernel-Level L2 Software Switch

VALE 是一个利用 netmap API 构建的**内核级 L2 学习型以太网交换机**。多个 netmap 客户端（或物理端口）可以连接到同一个 VALE 交换机实例，实现超高速的包交换。

VALE is a **kernel-level L2 learning Ethernet switch** built using the netmap API. Multiple netmap clients (or physical ports) can connect to the same VALE switch instance, achieving ultra-high-speed packet switching.

```
       ┌───────────────┐
       │  VALE switch  │
       │   "vale0"     │
       ├───┬───┬───┬───┤
       │   │   │   │   │
    ┌──┘   │   │   │   └──┐
    ▼      ▼   ▼   ▼      ▼
 [app1] [app2] [eth0] [VM guest]
```

**VALE 学习桥的工作原理：** / **How the VALE learning bridge works:**

1. 当交换机收到一个包，检查源 MAC 是否已在转发表中 / When the switch receives a packet, check if the source MAC is already in the forwarding table
2. 如果不在 → 记录（源 MAC → 接收端口）/ If not → record (source MAC → receiving port)
3. 查目标 MAC 的转发表 / Look up the destination MAC in the forwarding table
4. 如果找到 → 单播到目标端口 / If found → unicast to the destination port
5. 如果未找到（或目标 MAC 是广播）→ 泛洪到所有其他端口 / If not found (or destination MAC is broadcast) → flood to all other ports

**编译控制：** / **Compile control:**

```bash
./configure --enable-vale    # 启用 VALE（默认）/ Enable VALE (default)
./configure --disable-vale   # 禁用 VALE / Disable VALE
```

### 2. Pipe（默认启用）—— 背靠背虚拟端口对 / Pipe (Enabled by Default) — Back-to-Back Virtual Port Pair

Netmap pipe 是一对**背靠背连接**的虚拟 netmap 端口：一端发送的数据包从另一端直接接收，不经过任何网络设备。

Netmap pipe is a pair of **back-to-back connected** virtual netmap ports: packets sent from one end are directly received at the other end, without traversing any network device.

```
┌──────────┐         ┌──────────┐
│  App A   │────────▶│  App B   │
│ (pipe A) │◀────────│ (pipe B) │
└──────────┘         └──────────┘
       零拷贝内核级管道（类似双向 pipe）
       Zero-copy kernel-level pipe (like a bidirectional pipe)
```

**用途：** / **Use cases:**
- 进程间通信（IPC）/ Inter-process communication (IPC)
- 在独立进程中测试网络协议栈（一个进程产生流量，另一个接收并验证）/ Test network protocol stacks in separate processes (one generates traffic, the other receives and validates)
- QEMU/KVM 虚拟机互联（通过 netmap pipe 后端）/ QEMU/KVM VM interconnection (via netmap pipe backend)

### 3. Monitor（默认启用）—— 零拷贝端口监控 / Monitor (Enabled by Default) — Zero-Copy Port Monitoring

Netmap monitor 可以在**不中断流量**的情况下监控其他 netmap 端口：

Netmap monitor can monitor other netmap ports **without disrupting traffic**:

- **复制模式：** 将数据包的拷贝发送给监控客户端 / **Copy mode:** Sends a copy of the packet to the monitoring client
- **零拷贝模式：** 将数据包指针共享给监控客户端（更快，但原始客户端和监控客户端必须协调缓冲区所有权）/ **Zero-copy mode:** Shares packet pointers with the monitoring client (faster, but the original client and monitoring client must coordinate buffer ownership)

**应用场景：** / **Application scenarios:**
- 实时数据包分析（类似 tcpdump 但零拷贝）/ Real-time packet analysis (like tcpdump but zero-copy)
- 性能调试（监控流量模式而不影响被测系统）/ Performance debugging (monitor traffic patterns without affecting the system under test)
- 安全审计（在网络边界做被动流量检查）/ Security auditing (passive traffic inspection at network boundaries)

### 4. Generic（默认启用）—— 通用适配器 / Generic (Enabled by Default) — Generic Adapter

Generic 适配器允许**任何已有 Linux 驱动**的网络接口以 netmap 模式打开，但性能低于原生 netmap-aware 驱动。

The Generic adapter allows **any network interface with an existing Linux driver** to be opened in netmap mode, but with lower performance than native netmap-aware drivers.

```bash
# 任何网卡都可以通过 generic 适配器以 netmap 模式打开
# 不需要驱动补丁——但性能会显著下降
# Any NIC can be opened in netmap mode via the generic adapter
# No driver patch needed — but performance will be significantly lower
./pkt-gen -i eth0    # eth0 无需特殊驱动即可用 netmap 模式 / eth0 works in netmap mode without a special driver
```

**工作原理：** Generic 适配器在 netmap 的缓冲区与内核的 sk_buff 之间做双向拷贝。每次收发仍有一次内存拷贝操作，但批处理和预分配缓冲区仍然有效。

**How it works:** The Generic adapter performs bidirectional copying between netmap buffers and kernel sk_buffs. There is still one memory copy per send/receive, but batching and pre-allocated buffers remain effective.

### 5. ptnetmap（默认启用）—— 虚拟机透传 / ptnetmap (Enabled by Default) — VM Passthrough

ptnetmap 允许将 netmap 端口**透传（pass-through）**到 QEMU/KVM 虚拟机内部，guest 内的应用程序可以直接使用 netmap API。

### 驱动补丁的作用 / Purpose of Driver Patches

Netmap 补丁对驱动的主要修改：

The main modifications made by netmap patches to drivers:

4. **中断优化** —— 在 netmap 模式下调整中断节流以减少开销 / **Interrupt optimization** — Adjust interrupt throttling in netmap mode to reduce overhead

## 注意事项 / Notes

- **需要 root 权限：** 加载内核模块、打开 netmap 设备、操作 DMA 缓冲区都需要 root 权限 / **Root privileges required:** Loading kernel modules, opening netmap devices, and manipulating DMA buffers all require root privileges
- **网卡独占：** netmap 模式下的网卡不能同时被内核协议栈使用（IP 地址仍可配置，但数据包不会经过协议栈） / **NIC exclusivity:** A NIC in netmap mode cannot be used simultaneously by the kernel protocol stack (IP addresses can still be configured, but packets will not go through the protocol stack)
- **驱动冲突：** 如果加载了多个同名驱动（如标准 ixgbe + netmap 版 ixgbe），内核会选择先加载的 / **Driver conflicts:** If multiple drivers with the same name are loaded (e.g., standard ixgbe + netmap-patched ixgbe), the kernel selects the one loaded first
- **通用适配器性能：** 使用 Generic 模式（未打补丁的驱动）时，性能比 netmap-aware 模式低 3-10 倍 / **Generic adapter performance:** Using Generic mode (unpatched driver) results in 3-10x lower performance than netmap-aware mode
- **内核版本兼容性：** netmap 最稳定的支持范围是 Linux 4.x 和 5.x。6.x 内核可能需要手动调整补丁 / **Kernel version compatibility:** netmap is most stably supported on Linux 4.x and 5.x. 6.x kernels may require manual patch adjustments
- **中断处理：** 在 netmap 模式下，驱动通常禁用或节流中断，改用轮询模式（polling）——这意味着空闲时 CPU 使用率会升高 / **Interrupt handling:** In netmap mode, drivers typically disable or throttle interrupts and switch to polling mode — this means CPU usage increases when idle
- **VALE MAC 学习表大小：** 默认为 1024 条，大型拓扑中可能需要调整（重新编译内核模块） / **VALE MAC learning table size:** Default is 1024 entries, may need adjustment in large topologies (requires kernel module recompilation)
- **安全考虑：** `/dev/netmap` 设备节点的访问权限应严格控制（netmap 客户端可读写网卡内存中的任何数据包） / **Security considerations:** Access permissions for the `/dev/netmap` device node should be strictly controlled (netmap clients can read/write any packet in NIC memory)

### netmap_ring —— 环形缓冲区描述符 / Ring Buffer Descriptor

```c
struct netmap_ring {
    int64_t         buf_ofs;    // 缓冲区区域相对于映射起点的偏移 / Buffer area offset relative to mapping start
    uint32_t        num_slots;  // 环中的 slot 数量（通常 1024）/ Number of slots in ring (typically 1024)
    uint32_t        nr_buf_size;// 每个缓冲区的字节数（通常 2048）/ Bytes per buffer (typically 2048)
    uint16_t        ringid;     // 环索引（0-N）/ Ring index (0-N)
    uint16_t        dir;        // TX 或 RX / TX or RX
    int32_t         head;       // ★ 用户态写入（TX环中放置新包的位置；RX环中消费包的位置）/ Userspace writes (position for new packets in TX ring; consumption position in RX ring)
    int32_t         cur;        // ★ 内核写入（下一个要处理或已发送的 slot）/ Kernel writes (next slot to process or already sent)
    uint32_t        flags;      // 时间戳等标志 / Flags such as timestamps
    struct netmap_slot slot[];  // ★ 可变长度数组：数据包描述符 / Variable-length array: packet descriptors
};
```

```c

    uint16_t flags;     // NS_BUF_CHANGED（缓冲区内容已更新）/ (buffer content updated), NS_MOREFRAG（分片）/ (fragmentation)
    uint64_t ptr;       // 保留/时间戳（取决于 ring flags）/ Reserved/timestamp (depends on ring flags)
};
```

## 附录 D：线速计算参考 / Appendix D: Line Rate Calculation Reference

线速 pps 计算公式：/ Line rate pps calculation formula:

```
pps = line_rate / (672 + 8 * pkt_size)
```

其中 `line_rate` 为标称链路速率（bps），`pkt_size` 为实际包大小（含 MAC 头 + CRC）。

Where `line_rate` is the nominal link rate (bps) and `pkt_size` is the actual packet size (including MAC header + CRC).

|-----------|------|-----|------|------|

| 128 | 0.0589 Mpps | 0.589 Mpps | 5.89 Mpps | 23.58 Mpps |
